http缓存

发表于 2019-08-03 | 更新于 2019-09-19

缓存分类

强缓存

  • 强缓存不会向服务器发送请求,直接从缓存中读取数据.Chorm控制台会显示from disk cache 或 from memory cache ,返回200状态码

    协商缓存

  • 协商缓存向服务器发送请求,服务器根据请求头来判断是否命中缓存,若命返回 304 状态码

相关头

强缓存(Cache-Control/Expires)

  1. Expries
    服务器在 response header 返回这个过期时间,如果在这个过期时间内,则命中缓存
  2. Cache-Control
    no-cache:不缓存过期的资源,缓存会向服务器进行有效期确认
    no-store:不进行缓存

    Cache-Control优先级高于Expries,Cache-Control是http1.1的产物,Expries是http1.0的产物

协商缓存

  1. ETag和if-None-Match
    ETag是上一次文件加载时,服务器返回的,相当于资源文件的指纹,是资源的唯一标识.
    1. 浏览器在下一次资源请求时,会把ETag值放到 request-header 中的 if-none-match中
    2. 服务器拿到 ETag 值后与该资源文件的ETag值进行比对,若一样,则表示资源未更改,命中缓存,返回304状态码
  2. Last-Modified和if-Modified-Since
    Last-Modifed是表示资源最后一次更改时间.命中缓存过程与ETag类似
    1. 服务器会在 response-header 中返回 Last-Modified.
    2. 浏览器进行下一次资源请求时,会把 Last-Modified放到request-header中的 if-Modified-Since
    3. 服务器拿到 浏览器传过来的 Last-Modified,做对比,若相同则命中缓存
  • ETag在精度上要优于Last-Modified,Last-Modified时间单位是秒,若在一秒内更改多次,Last-Modified则体现不出多次更改而ETag会对每一次更改都做记录
  • 在性能上,ETag会略逊于Last-Modified,Last-Modified只是记录最后更改的时间,ETag是需要服务器通过算法计算出一个Hash值
    在优先级上,服务器校验ETag优先于Last-Modified

浏览器

  1. 浏览器第一次加载资源,服务器返回200,浏览器会把资源文件下载下来,并把 response header及该请求的返回时间一并缓存
  2. 下一次加载资源时,先比较当前时间与上一次返回200时的时间差,若没有超过Cache-control中的max-age,则没有过期,命中缓存.
  3. 如果过期,则向服务器发送header带有 if-None-Match 和 if-Modified-Since请求
  4. 服务器收到请求后,优先根据ETag的值判断请求文件是否有更改,如果ETag值一致则没有修改,命中协商缓存,返回304.如果不一致,则表示有改动,直接返回新的资源的新的ETag值
  5. 如果服务器收到的请求中没有ETag,则将if-Modified-Since与请求文件最后修改的时间做对比,若一致则命中缓存,返回304.不一致则返回新的资源文件和Last-Modified值,状态码200
